⏺ Voici mon analyse complète du projet Social Vision Agents.

  ---
  1. Le besoin métier

  Automatiser de bout en bout la production hebdomadaire de contenu social media pour une startup tech (produits IA/MLOps), sur 4 plateformes (LinkedIn, X, TikTok, Facebook), avec :

  - Plusieurs campagnes produit par semaine (nombre de posts dynamique)
  - Des posts "vision/thought leadership" non liés à un produit
  - Une recherche web en temps réel pour enrichir le contenu
  - Une validation humaine avant publication
  - Une persistance complète en PostgreSQL

  State partagé (AgentState — TypedDict)

  Un "whiteboard" unique (src/state.py) avec checkpoint LangGraph à chaque étape. Contient : messages, current_week, vision_summary, research_insights, calendar, product_spotlights, drafts,
  approved_drafts, report, error.

  Stack LLM avec fallback

  Ollama (Llama 4 local) → Groq (cloud) → Gemini (cloud)

  La factory (src/utils/llm_factory.py) fait un ping au provider primaire et bascule automatiquement sur le fallback si échec.

  Persistance PostgreSQL

  - Table weekly_batches : un row par semaine (JSONB contenant tous les posts)
  - Table posts : chaque post individuel avec historique de statut, métriques d'engagement
  - Driver asyncpg avec connection pooling
  - Upsert sur (week, platform, day)

  Système de Skills

  7 fichiers SKILL.md dans skills/public/ (social-media-writing, viral-hooks, hashtag-strategy, visual-branding, etc.) avec frontmatter YAML. Injectés dynamiquement dans les prompts agents selon le rôle
  (src/skills/loader.py).

  API FastAPI (main.py)

  - POST /run — lance un cycle complet
  - POST /approve/{week} — reprend après l'interruption humaine
  - GET /status/{week} — état du run
  - GET /posts/{week} — posts stockés en PostgreSQL
  - POST /images/upload/{week} — upload d'images manuelles
  - GET /skills — catalogue des skills

  Infrastructure Docker

  4 services : ollama (GPU), postgres:16, app (FastAPI), scheduler (APScheduler, cron lundi 8h).

  4. Points forts du design

  1. Zéro hardcoding — tout est piloté par weekly_brief.yaml. Changer les campagnes, ajouter un produit, modifier le nombre de posts : aucune modification de code.
  2. Images fournies, pas générées — évite la complexité et le coût de la génération d'images. Les URLs sont dans le YAML par campagne/plateforme.
  3. Crash recovery — checkpoint LangGraph à chaque nœud + human interrupt propre (reprise avec le même thread_id).
  4. Multi-provider LLM — pas de vendor lock-in, bascule automatique Ollama → Groq → Gemini.
  5. Recherche web — Tavily + DuckDuckGo pour du contenu ancré dans l'actualité.

  ---
  5. Points d'attention / axes d'amélioration potentiels
  Sujet: Checkpointer
  Observation: MemorySaver() en dev — pas de persistance des checkpoints entre redémarrages. En prod il faudrait PostgresSaver.
  ────────────────────────────────────────
  Sujet: Pas de publication réelle
  Observation: Les tokens d'API (LinkedIn, X, TikTok, Meta) sont configurés mais aucun code de publication effective n'existe dans publisher.py — le nœud ne fait qu'exporter en fichiers et sauvegarder en

    BDD.
  ────────────────────────────────────────
  Sujet: Pas de parallélisme
  Observation: Les 8 nœuds sont strictement séquentiels. Le writer et le visual pourraient potentiellement traiter les drafts en parallèle par plateforme.
  ────────────────────────────────────────
  Sujet: save_all_posts séquentiel
  Observation: Sauvegarde post par post au lieu d'un batch INSERT — impact en performance si beaucoup de posts.
  ────────────────────────────────────────
  Sujet: Pas de retry LLM par nœud
  Observation: Si un nœud échoue à parser la sortie JSON du LLM, l'erreur est loggée mais il n'y a pas de retry automatique.
  ────────────────────────────────────────
  Sujet: Tests
  Observation: La structure existe (tests/unit/, tests/integration/) mais le contenu est à vérifier.
  ---
  6. Synthèse du besoin

  Ce projet est un système multi-agents de production de contenu social media, conçu pour fonctionner en production avec :

  - Un pipeline LangGraph séquentiel à 8 étapes avec checkpoint et interruption humaine
  - Un LLM local-first (Ollama/Llama 4) avec fallback cloud (Groq, Gemini)
  - Une configuration déclarative (YAML) qui pilote tout le comportement hebdomadaire
  - Une persistance PostgreSQL complète
  - Un système de skills injectables pour enrichir les capacités de chaque agent
  - Un déploiement Docker Compose prêt pour la production

  Le système génère du contenu, le soumet à validation humaine, puis le sauvegarde — mais ne publie pas encore automatiquement sur les réseaux sociaux.
